### 1. `apps/web-pc`
This is a sleek, premium IPTV application built with **React, Vite, and Electron**. Designed to provide a modern, cinematic viewing experience mimicking top-tier streaming services (Netflix, Hulu). It features advanced HLS streaming proxies and real-time FFmpeg AC3 audio transcoding.

*For detailed instructions on the PC App, read: [`apps/web-pc/README.md`](./apps/web-pc/README.md)*

#### PC App Features:
- **Zero-Downtime EPG Hot-Reload**: A background Node.js service watches the `StreamPro EPGs` folder for new XML exports. When detected, an off-thread Worker streams the massive XML, pre-calculating ISO dates into standard UNIX timestamps and offloading Base64 encoding from the main CPU. It then strips the payload of all XML bloat before executing the IPC transfer, fully preventing V8 "Out of Memory" crashes on 1GB+ files. Because the main Express API now serves pre-computed integers instead of running thousands of synchronous Regex evaluations on-the-fly, frontend EPG hydration API requests resolve in ~0ms. The background service pushes a Server-Sent Event (SSE) to seamlessly inject the newly parsed data into the React UI without dropping a single 60fps video frame.
- **Extreme V8 Engine & Rendering Optimizations**: The React Virtual DOM has been rigorously optimized for memory and speed. The EPG Grid employs a highly tuned `for` loop break algorithm to surgically cull thousands of off-screen programs and React components during horizontal scrolling, dropping DOM element calculations from 4,500+ down to ~300 per frame. Data parsing bottlenecks (like `.includes()` loops on 70,000 item arrays) were completely rebuilt using pre-compiled native Regular Expressions and persistent memory caches. Additionally, the backend Node.js Image Proxy bypasses main-thread locking by using an in-memory Metadata map and asynchronous filesystem access (`fs.promises`), allowing the frontend to smoothly stream 100+ channel logos concurrently while scrolling without ever dropping frames.
- **Multi-View System**: Advanced picture-in-picture and split-screen engine supporting up to 4 simultaneous live streams. Includes frictionless selection flows like an in-player Quick Split HUD, global right-click Context Menus, and a dedicated EPG Setup Mode.
- **Native Catch-Up TV (Time-Travel)**: Built-in Time-Travel functionality inside the TV Guide header, featuring quick-jump buttons to smoothly scroll the EPG timeline up to 72 hours backward. This UI seamlessly unlocks the backend Xtream &timeshift= API, allowing users to instantly play archived TV shows natively in the video player without requiring manual DVR recordings.
- **Cinematic Player**: Custom React video player with native HLS and MP4 fallback, handling embedded VTT/SRT subtitles and multi-track audio. Includes options for Auto-Play Next Episode for TV Series, an "Are You Still Watching?" idle timeout (Configurable in Settings), and a customizable Sleep Timer.
- **TMDB Metadata Enrichment & Image Proxy**: Automatically fetches high-resolution posters from TMDB for missing IPTV media. If TMDB fails, it forces the IPTV provider's original image through a local Node.js proxy (injecting a `VLC` User-Agent) to bypass strict 403 Forbidden hotlink blocks.
- **Procedural Dynamic Posters**: When a title is too obscure to exist on TMDB or the IPTV provider, the app dynamically generates a premium, moody cinematic gradient poster driven by a string-hash of the movie's title, ensuring the UI never breaks immersion with "blank" missing images.
- **Advanced Multi-View Grid**: Seamlessly add up to 4 Live TV channels to a dynamically scaling split-screen grid. The grid automatically shrinks into Picture-in-Picture mode while browsing the EPG to add more channels, and features a unified Global Master Volume synced precisely to the actively focused channel (Primary Audio).
- **Persistent Global Volume**: The video player leverages `localStorage` and a central Zustand state store to ensure that your custom volume settings natively survive stream hot-swaps, channel zapping, and application reloads.
- **My Library / Favorites Manager**: Native HTML5 Drag-and-Drop library for organizing favorited Channels, Movies, and Series into custom saved sequences, alongside Alphabetical sorting.
- **Auto-Tuning & DVR Reminders**: Global background daemon (`LayoutManager.jsx`) that constantly monitors user-scheduled broadcasts, triggering native OS push notifications, custom UI toast banners, or automatically launching live feeds exactly at program start.
- **Smart EPG Hiding Engine**: An intelligent heuristic filter that automatically hides empty dead-spot channels from the guide, while mathematically preserving static content categories (like 24/7 Networks, PPV, and Sports).

### 2. `apps/epg-editor`
This is a standalone Native Desktop Application built with **React, Vite, and Electron** specifically designed to handle importing, parsing, and mapping massive community EPG XMLTV databases (up to 536MB+). It produces a compact `my_epg.xml` file on the user's Desktop containing only the channels that are actually in their M3U playlist â€” stripping 99% of unnecessary schedule data that causes memory crashes in the main `web-pc` app.

#### EPG Editor â€” How It Actually Works
After extensive debugging (see changelog), the working architecture is as follows:

**Step 1 â€” M3U Ingest (Live TV Only)**
The user uploads their provider `.m3u` file. The server-side parser reads every `#EXTINF:` line and applies a **dual-signal VOD filter** before accepting a channel:
- Checks `group-title` attribute for VOD keywords (`movie`, `series`, `vod`, `episode`, `season`, `film`, `anime`, `xxx`)
- Checks the stream URL for static file extensions (`.mp4`, `.mkv`, `.avi`, etc.)

A full provider M3U typically contains 490,000+ entries (Live + VOD + Series). After filtering, only ~9,000â€“10,000 live channels remain. This is critical â€” running the Jaccard matcher on 490,000 entries would take hours; on 10,000 it takes 2â€“3 minutes.

**Step 2 â€” Community EPG Download & SQLite Pipeline**
The user selects one or more regional XMLTV databases from `epgshare01.online`. Each file is processed sequentially:
1. `axios` streams the `.gz` file to disk (never loaded into RAM)
2. `zlib.createGunzip()` pipes directly to a temp file via Node.js stream pipeline
3. A `readline` interface reads the unzipped XML **line by line** â€” never buffering the full file
4. Individual `<channel>` and `<programme>` blocks are assembled line by line, then parsed with `fast-xml-parser` (each block is ~200â€“500 bytes, not 536MB)
5. Programme records are inserted into a local **SQLite database** in batches of 50,000, committed with explicit `BEGIN TRANSACTION` / `COMMIT` cycles

Progress is broadcast to the React UI via **Server-Sent Events (SSE)** so the user sees live schedule counts updating without the HTTP connection timing out.

**Step 3 â€” Jaccard Auto-Matching**
Once all EPG files are parsed, the auto-map engine runs a multi-pass channel resolution pipeline. Passes are attempted in order; a channel exits the loop as soon as any pass succeeds.

- **Pass 0 (AI Override Map)**: Loads hardcoded 1:1 mappings from `manual_overrides.json`. Now features a **Normalized Override Dictionary** that automatically makes matches completely immune to spacing, punctuation, and casing differences in the provider's M3U feed. It also natively supports overriding raw `tvg-id`s directly, guaranteeing 100% accuracy for problem edge cases and hyper-local affiliates before any algorithmic logic runs.
- **Pass 0a â€” Direct tvg-id lookup (O(1))**: Looks up `ch.tvgId` directly in `channelIdMap` (a dictionary built from every EPG `<channel id="...">` value). E.g. M3U `tvg-id="amc.us"` â†’ EPG `channel id="amc.us"` â†’ instant match. Resolves ~70% of channels.
- **Pass 0b â€” tvg-name direct lookup**: Same O(1) lookup against the tvg-name field. Some providers use tvg-name as the EPG ID.
- **Pass 0c â€” tvg-id + country suffix**: If the tvg-id has no `.` (bare abbreviation like `AMC`, `TBS`), tries appending `.us`, `.uk`, `.ca`, `.au`, `.nz`, `.ie`, `.za` in order. Catches providers that omit the country suffix from their tvg-id.
- **Pass 1:** Exact display-name match (O(1)).
- **Pass 1.5:** Substring / Prefix match. Leverages `Array.includes()` to map complex names that completely contain the core identity (e.g., matching `USA Buzzr HD` to `BUZZR STREAM`). Mathematically prevents highly generic names (like `Discovery+`) from falsely overriding longer, more specific targets by always prioritizing the longest character overlap.
- **Pass 2:** Jaccard similarity fallback (`jaccardSim`). Scans bigram intersections to resolve misspellings and subtle structural differences (e.g. `Hallmark Mystery` vs `Hallmark Movies And Mysteries`). Automatically lowers strictness thresholds for channels that share identical 4-letter core prefixes.
- **Geographic Boundary Guardrails:** During all fuzzy matching (Pass 1.5 and 2), the algorithm strictly parses the M3U channel for geographic identity (e.g. `USA`, `UK`, `CA`). If detected, it mathematically locks the channel out of matching any EPG target assigned to a differing region (e.g., `USA Discovery` cannot match `DiscoveryChannel.au`). This explicitly scans the *entire* array of a target EPG item's IDs to ensure fallback extensions (like `.us2` or `.us_locals`) correctly validate, eliminating cross-border false positives without accidentally dropping valid USA channels. North American callsigns matched during Pass 1b explicitly bypass this guard because they are globally unique.
- **Pass 3 â€” tvg-id propagation**: After the main loop, builds a `tvgId â†’ epgId` lookup from all matched channels, then copies those assignments to all still-unmatched channels that share the same tvg-id. Automatically resolves East/West/UHD/FHD/LHD duplicate variants.
- **Pass 3b â€” Base Name propagation**: Rescues remaining East/West variants lacking a `tvg-id` by matching their stripped base names against the base names of already successfully matched channels.

### 4. `apps/tv-app` (Upcoming)
This will be the Native Android TV and Amazon Firestick application built with **React Native**. Because low-end devices like Firesticks only possess ~1.5GB of RAM, we cannot use the PC architecture (Node.js/Chromium). The TV app will rely on three strict architectural pillars:
- **Direct Play (ExoPlayer)**: We will bypass browser media engines completely. By implementing native Android `ExoPlayer`, the app will natively decode `.ts` firehose streams, AC3 Audio, and HEVC/H.265 video with zero local proxy transcoding required, achieving sub-1-second channel zapping.
- **Native Background Threading (Java/Kotlin)**: We cannot parse massive 500MB XMLTV payloads on the single React Native Javascript thread without freezing the UI for 5+ minutes. All EPG downloading and fuzzy-matching logic will be shifted into a custom Native Android module that runs silently on a background thread.
- **SQLite EPG Persistence**: Instead of holding 15,000 channels and programs in active RAM (which causes instant Out-of-Memory crashes on Firesticks), the native module will stream the parsed XML data directly into a local device **SQLite Database**. The React UI will query this database only for the ~10 channels currently visible on the screen, dropping active memory consumption from 800MB to ~15MB.

### ðŸ–¥ï¸ Browser vs. Desktop Environment (Important)
While you can run the app in a standard Chrome browser using `npm run dev`, it is **highly recommended** to test and develop using the Native Desktop Environment (`npm run start:desktop`). Running the app inside the Electron wrapper permanently bypasses strict browser-level CORS security sandboxes. This guarantees that direct live IPTV video streams, remote HLS chunks, and external TMDB images will load flawlessly and instantly without encountering cross-origin network blocks. Furthermore, standard web browsers struggle with massive DOM updates; we previously had to artificially limit (`.slice()`) the VOD and Channel arrays to prevent browser crashes. The high-performance Electron desktop environment handles these massive data arrays with ease, so artificial array slicing must remain disabled to show the full library.
