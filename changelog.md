# StreamPro IPTV - Changelog

## [2026-06-27] EPG Architecture Overhaul: Exact Matching & Cloud Workflows
- **`process_epg.js`**: Enhanced the SAX XML parser to correctly extract `<display-name>` tags and map them directly to their parent `<channel id="...">`. These exact mappings are now exported to a new `display_names.json` file.
- **`process_epg.js`**: Fixed a critical memory leak during SAX parsing where `currentTag` was not being cleared on `closetag`, causing rogue text nodes (like whitespace) to overwrite mappings during processing of large (600MB+) XML files.
- **`api.js`**: Completely rewrote `getCustomEpgBulk` to eliminate unreliable "fuzzy guessing" and instead use a robust, deterministic 5-tier fallback system:
  1. **Manual Overrides**: Case-insensitive check of user mappings from `epg-editor`.
  2. **Display Names**: Uses the new `display_names.json` to perfectly link provider channel names (e.g. "USA AMC") to the XML `<channel id="...">`.
  3. **Exact EPG ID**: Case-insensitive matching of the provider's `epg_channel_id`.
  4. **Generated Channel Prefix**: Support for custom generated XMLs by testing `channel_${stream_id}`.
  5. **Ultimate Fallback**: Raw ID.
- **`.github/workflows/update_epg.yml`**: Added `--max-old-space-size=4096` to the `node` execution command to prevent GitHub Actions from crashing with `Out Of Memory` errors when chewing through 600MB+ XML payloads. Added instructions for users to provide their `EPG_URL` as a GitHub Secret so the cloud runner automatically downloads and processes the massive files rather than relying on direct repository uploads (which bypasses GitHub's 100MB file size limit).
## [2026-06-27] Critical Fix: EPG Broadcasts Were Silently Failing
- **`server.js`**: `broadcastEpgUpdated()` and `epgSseClients` were declared at lines ~1410/1384 but called from `spawnEpgWorker` at line ~1048. `const` is NOT hoisted — every SSE broadcast was throwing a `ReferenceError` silently inside the worker's message handler, meaning the frontend **never received the signal to fetch EPG data**. Fixed by moving both declarations before `spawnEpgWorker`.
- **`server.js`**: Removed orphaned `} catch (workerErr)` block left by a previous edit — was causing a `SyntaxError: Unexpected token 'catch'` crash on startup. Restored proper `try { await spawnEpgWorker(url); } catch` structure.
- Both files now pass `node --check` syntax validation.

## [2026-06-27] Critical Fix: EPG Database Crash and Scrolling Lag
- **`EPGGrid.jsx`**: Fixed a critical bug where the UI would pass the entire category (up to 15,000+ channel IDs) into a single IndexedDB `.anyOf()` query via `useLiveQuery`. This instantly crashed the database transaction and caused the "No Program Information Available" error. Refactored the component to virtualize the query, only fetching EPG data for the ~30 channels currently visible on screen. Additionally added a 150ms debounce to the visible channels calculation so rapid scrolling no longer spams the database, completely eliminating UI stutter and lag.
- **`ModernTheme.jsx`**: Cleaned up legacy `epgData` prop drilling that evaluated to `undefined` after the previous IndexedDB migration.
- **`useAppStore.js` & `epgDatabase.js`**: Added a new `purgeStaleEpgPrograms()` background task that automatically sweeps the database and deletes TV guide data older than 48 hours to prevent infinite storage bloat.
- **`useAppStore.js`**: Fixed a massive UI freeze when opening the EPG. Restored the 100-channel chunking logic during the initial `bulkPut` database insertion loop.
- **`useAppStore.js`**: Fixed a critical "No Program Information Available" bug across the entire EPG grid and resolved the issue where switching categories resulted in an empty guide. A previous performance patch attempted to speed up initial loading by directly reading the backend `epg_cache.json` using Electron IPC instead of calling the `/api/custom-epg/bulk` endpoint. However, the raw server cache stores programs by `epg_channel_id` rather than the `stream_id` expected by the frontend's IndexedDB query component, causing silent key collisions. Removed this broken IPC shortcut, restoring the `custom-epg/bulk` network call. Furthermore, modified the background fetching loop to request EPG mappings for *all global channels* simultaneously rather than just the active category. This guarantees that when a user switches to a new category, the guide is instantly populated instead of permanently blanking out.
- **`EPGGrid.jsx`**: Fixed a major bug where scrolling down and back up would cause all TV guide information to disappear. This was caused by the `useLiveQuery` hook constantly dropping its Dexie database subscription when the visible channel parameters changed too quickly. Refactored the data layer to use a permanent `epgCacheRef` memory cache. Now, once a channel's EPG data is loaded into memory, it is retained permanently for that session, resulting in zero flashing and flawlessly smooth scrolling up and down the grid.

## [2026-06-27] Progressive EPG Population (No More Waiting)
- **`epgWorker.js`**: Worker now sends partial batches every 3,000 programs via `parentPort.postMessage({ type: 'batch', ... })` instead of waiting until full parse is done. First batch fires within 15-30 seconds of startup.
- **`server.js`**: `spawnEpgWorker()` now handles streaming `batch` messages — merges incoming programs into `cachedCustomEpg` as they arrive and broadcasts `epg_updated` SSE after the first batch so the frontend starts fetching immediately. Final `done` message triggers a disk-reload and second broadcast to fill in remaining channels.
- **`useAppStore.js`**: SSE handler now uses additive `fetchEpgAdditive()` which writes programs to Dexie via `bulkPut` without clearing the database first. This means each SSE event adds more data progressively. A separate `epg_full_reset` event is reserved for manual force-refresh (wipe + re-fetch).
