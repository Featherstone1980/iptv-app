# EPG Architecture & Exact Matching Blueprint

This document explains the custom EPG (Electronic Program Guide) architecture for the StreamPro IPTV App. It covers how we achieve instant load times on low-power devices, how the matching logic works without guessing, and how the internal micro-batching engine is structured so it can be rebuilt if ever lost.

## 1. The EPG Guide Pipeline: How Data Moves

To prevent the frontend browser and backend Node.js server from crashing under the weight of 600MB+ XML files (containing millions of programs), the EPG system uses a heavily decoupled **Micro-Batching + IndexedDB Pipeline**.

### Step 1: Backend Parsing (epgWorker.js)
When the Node.js server starts, an off-thread worker (epgWorker.js) loads the XML EPG file into RAM. It pre-calculates ISO timestamps into Unix integers and creates a highly optimized epgData map. This keeps the massive data out of the V8 main thread.

### Step 2: Frontend Micro-Batching (useAppStore.js)
When the React frontend initializes, it possesses an array of ~9,000 live channels. It **cannot** request the EPG for all 9,000 channels simultaneously, or the server will crash and the browser will run out of memory. 
- The frontend slices the 9,000 channels into **Micro-Batches of 100 channels**.
- It sequentially POSTs each batch of 100 to the backend (/api/custom-epg/bulk).

### Step 3: Fast Jaccard Matching (server.js)
The backend receives 100 channels. It attempts to match each channel to the EPG XML data using a strict 4-Tier fallback system:
1. **Manual Overrides**: Checks uzzyMatchCache (generated by the standalone EPG Editor).
2. **Exact EPG ID**: Checks if the provider's epg_channel_id is an exact match.
3. **Exact Name Match**: Fast dictionary lookup against a cleaned, alphanumeric-only channel name.
4. **Fuzzy Jaccard Match**: Uses a mathematical Bigram Jaccard similarity algorithm (>= 0.6 threshold) to match misspellings. 
   > **CRITICAL ARCHITECTURE RULE**: The Jaccard loop must **NOT** create arrays inside the loop. Object.entries(epgData.channelMap) must be **hoisted** outside the 100-channel loop. If not hoisted, the server generates over 324,000 arrays per batch, causing catastrophic garbage collection stalling (increasing response times from 5ms to 5000ms+).

### Step 4: IndexedDB Persistence (Dexie.js)
When the frontend receives the matched programs for the 100 channels, it does **not** store them in React State or a standard Javascript Object (which causes OOM crashes). 
- It uses Dexie.js to insert the chunk directly into the browser's persistent IndexedDB storage using ulkPut().
- **CRITICAL ARCHITECTURE RULE**: After every ulkPut(), the frontend MUST yield to the main thread (wait new Promise(r => setTimeout(r, 100))). This 100ms yield prevents the massive DB inserts from locking the browser's UI thread, ensuring the app continues rendering video at 60fps while the EPG syncs in the background.

## 2. Aggressive Preload Scrolling (EPGGrid.jsx)

To achieve instant, zero-latency scrolling in the React Guide, the frontend completely decouples the **Render Window** from the **Database Query Window**.

- **The Render Window**: React only renders 15 rows above and below the current viewport to keep the DOM light (preventing layout thrashing).
- **The Preload Buffer**: Behind the scenes, a massive PRELOAD_BUFFER eagerly queries IndexedDB for **100 rows above** and **100 rows below** the viewport. 
- **The Debounce**: As the user scrolls, the scroll position updates constantly. A 30ms debounce timer prevents spamming the database on every pixel scrolled. Once the user stops scrolling for 30ms, the 200-channel preload window is passed to IndexedDB.
- **The Result**: Because the database fetches 100 channels *ahead* of the user's vision, aggressive vertical scrolling simply reveals rows that have *already* been loaded into the ultra-fast Javascript memory Map (epgCacheRef), guaranteeing instant renders without blank screens.

## 3. Dynamic Cache Invalidation

Because the background micro-batching takes about ~30-60 seconds to finish all 9,000 channels, the user might open the guide while it is still downloading.
- If a channel is visible on screen but its data hasn't arrived yet, the cache stores an "empty" array for it.
- A useEffect hook in EPGGrid.jsx actively listens to the epgLoadingProgress state. 
- Every time a new micro-batch finishes (progress increments), the UI aggressively scans the visible rows. If any row has an "empty" cache, it invalidates it, forcing it to immediately re-query IndexedDB to display the newly arrived data without waiting for a scroll event.
